## Деплой и CI (трогать деплой = риск для прода — на решение владельца)

- **CRITICAL** (codex) | `.github/workflows/deploy.yml:44` | Деплой делает `git pull` (движущийся master), а не checkout SHA, прошедшего тесты. Коммит A проходит CI, следом пушится B; деплой A подтягивает B, даже если тесты B падают/не закончились — тест-гейт обходится полностью. → Деплоить конкретный `github.sha` (checkout по SHA), не `pull`.
- **MAJOR** | `.github/workflows/deploy.yml` | Деплой зелёный, даже если бот умер на старте: `systemctl restart` возвращает 0 сразу после запуска процесса, не дожидаясь готовности. → Добавить после рестарта `sleep 5 && systemctl is-active --quiet robocat` (или проверку по journalctl), чтобы упавший старт красил деплой.
- **MAJOR** | `deploy.yml` | Неатомарный деплой: `git pull` до `pip install`. Упавший pip оставляет новый код без зависимостей → внеплановый рестарт/ребут = crash-loop на «новый код + старые deps». → Ставить зависимости из fetch'нутого коммита ДО переключения дерева, либо `git pull` только после успешного `pip install`.
- **MAJOR** | `requirements.txt` | Только нижние границы версий, лок-файла нет: CI тестирует свежие версии, прод живёт на замороженных от даты создания venv, пересборка venv может подтянуть мажорный breaking-релиз disnake/openai. → `pip-compile` lock или верхние границы у `disnake`/`openai`.
- **MINOR** | `deploy.yml` | Нет `concurrency`-группы и `timeout-minutes` у deploy-джобы: два быстрых пуша деплоятся параллельно (рестарт посреди чужого pip install), зависший SSH держит ран до 6ч. → `concurrency: {group: deploy, cancel-in-progress: false}` + таймаут.
- **MINOR** | `deploy.yml:33` | `appleboy/ssh-action@v1.0.3` — мутабельный тег с доступом к прод-SSH-ключу. → Пин по полному commit SHA.
- **MINOR** | `deploy.yml` | `pytest`/`pytest-asyncio` без пинов в CI: breaking-релиз инструмента блокирует все деплои, в т.ч. срочный хотфикс. → Пины `pytest~=8` `pytest-asyncio~=1`.
- **MINOR** | `deploy.yml` | Деплой не запускает `data/db_init.py` — новая таблица + использующий код деплоятся зелёными, но в рантайме `no such table`. → `python data/db_init.py` перед рестартом (идемпотентен).
